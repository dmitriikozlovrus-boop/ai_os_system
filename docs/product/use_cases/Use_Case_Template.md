# Use Case: Название

## Назначение шаблона

Этот шаблон используется для описания пользовательского сценария AI OS до передачи задачи в разработку.

Use case — это не кодовая задача.

Use case должен описывать:

- какую пользовательскую проблему решает сценарий;
- какой вход получает система;
- какой результат должна вернуть;
- какие сущности, сервисы, роли и правила задействованы;
- какие документы должен прочитать Codex перед реализацией;
- какие файлы можно и нельзя менять.

---

## 1. Цель

Кратко описать, зачем нужен use case.

Ответить на вопросы:

- какую проблему решает сценарий;
- для кого он нужен;
- какой практический результат должен появиться;
- почему этот сценарий важен для AI OS.

Пример:

```text
Цель use case — корректно раскладывать входящие пользовательские сообщения по нужным БД: Tasks, Events, Study, Problems, Ideas.
```

---

## 2. Пользовательский сценарий

Описать сценарий простым языком от лица пользователя.

Пример:

```text
Пользователь отправляет в Lyuba текстовое или голосовое сообщение.
Система должна понять, что это: задача, событие, исследование, проблема или идея.
После этого система должна создать или обновить запись в нужной БД и вернуть пользователю подтверждение.
```

---

## 3. Вход

Описать, что поступает в систему.

Указать:

- источник входа;
- формат входа;
- обязательные данные;
- возможные неполные данные;
- ограничения.

Пример:

```text
Источник: Lyuba / Telegram.
Формат: текстовое сообщение.
Возможные варианты: задача, событие, идея, проблема, исследовательский вопрос.
```

---

## 4. Выход

Описать, что система должна вернуть или создать.

Указать:

- пользовательский результат;
- системный результат;
- где должен быть виден результат;
- что делать при ошибке или неопределенности.

Пример:

```text
Система создает запись в нужной БД и возвращает пользователю подтверждение через Lyuba.
Если тип сущности не определен, система задает уточняющий вопрос.
```

---

## 5. Задействованные сущности

Перечислить сущности, которые участвуют в сценарии.

Примеры:

- Task;
- Event;
- Study;
- Problem;
- Idea;
- Project;
- Stream;
- Topic;
- Contact;
- Error;
- Source;
- Research Output.

Для каждой сущности указать роль в use case.

| Сущность | Роль в use case | Источник истины |
|---|---|---|
| Task | создается или обновляется | указать БД |
| Event | создается или обновляется | указать БД / Calendar |
| Error | создается при сбое | Error Log |

---

## 6. Задействованные агенты и роли

Указать только роли, реально участвующие в сценарии.

Не добавлять будущих агентов без необходимости.

| Роль | Статус | Задача в use case |
|---|---|---|
| Lyuba | Active | принимает вход и возвращает результат |
| Conductor | Active / Design | классифицирует вход и выбирает маршрут |
| Research Agent | Candidate | использовать только если нужен research |
| Task Agent | Candidate | использовать только если нужен интеллектуальный анализ задач |
| Calendar Agent | Candidate | использовать только если нужен интеллектуальный анализ календаря |

Важно:

```text
Если сценарий можно реализовать через Conductor + Core Logic + Integration Layer, отдельный агент не нужен.
```

---

## 7. Задействованные сервисы

Перечислить сервисы, инструменты и API, которые участвуют в сценарии.

| Сервис | Слой | Роль |
|---|---|---|
| Lyuba / Telegram | Input / Output Interface | прием входа и возврат результата |
| Notion | Data Layer | хранение записей |
| Todoist | Execution / Output | задачи |
| Google Calendar | Execution / Output | события |
| Gmail | Source / Execution | письма |
| OpenAI / GPT | Intelligence Layer | классификация, извлечение, генерация |

---

## 8. Источники данных

Описать, откуда система берет данные для выполнения сценария.

Примеры:

- сообщение пользователя;
- существующие записи в Notion;
- задачи в Todoist;
- события в Google Calendar;
- письма в Gmail;
- документы в Google Drive;
- правила из Knowledge;
- системные документы из GitHub.

Указать, какие источники являются источниками истины.

---

## 9. Правила обработки

Описать правила, по которым система должна обрабатывать вход.

Примеры правил:

- как определять тип сущности;
- когда создавать новую запись;
- когда обновлять существующую запись;
- когда задавать уточняющий вопрос;
- как обрабатывать неполные данные;
- как обрабатывать дубли;
- как фиксировать ошибки;
- какие поля обязательны;
- какие поля могут быть заполнены позже.

Пример:

```text
Если вход содержит конкретное действие, срок или ожидаемый результат, система должна рассмотреть вариант Task.
Если вход содержит дату, время, участников или место, система должна рассмотреть вариант Event.
Если вход содержит вопрос на изучение, система должна рассмотреть вариант Study.
```

---

## 10. Что входит в use case

Перечислить границы сценария.

Пример:

```text
Входит:
- прием входящего сообщения;
- классификация сущности;
- извлечение основных полей;
- выбор целевой БД;
- создание или обновление записи;
- возврат подтверждения пользователю;
- фиксация ошибки при сбое.
```

---

## 11. Что не входит в use case

Явно указать, что не должно делаться в рамках сценария.

Пример:

```text
Не входит:
- реализация всех будущих агентов;
- изменение общей архитектуры AI OS;
- создание новых БД без отдельного решения;
- изменение источников истины без обновления Data Ownership Map;
- разработка dashboard;
- автоматическая приоритизация всех задач.
```

---

## 12. Критерии готовности

Описать, как понять, что use case реализован.

Критерии должны быть проверяемыми.

Пример:

```text
Use case считается готовым, если:
- система корректно классифицирует тестовые входы;
- записи создаются в правильных БД;
- обязательные поля заполняются;
- неполные случаи приводят к уточняющему вопросу;
- ошибки фиксируются в Error Log;
- пользователь получает понятное подтверждение;
- Codex не изменил запрещенные файлы.
```

## 13. Тестовые примеры

Добавить минимальный набор тестовых входов и ожидаемых результатов.

| № | Вход | Ожидаемый тип | Ожидаемый результат |
|---|---|---|---|
| 1 | Напомни написать в CNBV завтра | Task | создана задача |
| 2 | Встреча с CANACINTRA 25 июня в 12:00 | Event | создано событие / запись |
| 3 | Изучить рынок биометрии в Мексике | Study | создана запись Study |
| 4 | Проблема: задачи теряются между Notion и Todoist | Problem | создана запись Problem |
| 5 | Идея: сделать дайджест по энергетике | Idea | создана запись Idea |

## 14. Риски

Описать основные риски реализации.

Примеры:

- неправильная классификация сущности;
- дублирование записей;
- потеря данных при синхронизации;
- конфликт источников истины;
- чрезмерное усложнение логики;
- преждевременное создание агентов;
- изменение архитектурных файлов без необходимости;
- слишком широкий scope для одного use case.

---

## 15. Открытые вопросы

Перечислить вопросы, которые нужно решить до реализации или в ходе проектирования.

Примеры:

- какая БД является источником истины для сущности;
- какие поля обязательны;
- когда задавать уточняющий вопрос;
- как обрабатывать дубли;
- какие статусы использовать;
- какие ошибки логировать;
- какие действия требуют подтверждения пользователя.

## 16. Документы, которые должен читать Codex перед реализацией

Перечислить документы, которые Codex должен изучить перед реализацией use case.

Минимальный набор:

```text
docs/architecture/System_Map.md
docs/architecture/System_Component_Registry.md
docs/architecture/Document_Placement_Rules.md
knowledge/agents/Agent_Registry.md

```

Дополнительно при необходимости:

```text
docs/conductor/Conductor_Service_Description.md
```

Правило:

```text
Codex не должен читать все документы подряд.
Codex должен читать только документы, релевантные конкретному use case.
```

## 17. Разрешенные файлы для изменения

Указать, какие файлы Codex может менять в рамках реализации.

Пример:

```text
Разрешено изменять:
- конкретный файл use case;
- файлы реализации, явно относящиеся к сценарию;
- тесты для данного сценария;
- документацию сценария;
- интеграционный файл, если это прямо указано в задаче.
```

Заполняется отдельно для каждого use case.

Пример для конкретного сценария:

```text
Разрешенные файлы:
- docs/product/use_cases/Entity_Routing_to_DB.md
- docs/product/use_cases/Tasks_Todoist_Sync.md
- src/core/task/*
- src/integrations/todoist/*
- tests/task_sync/*
```

---

## 18. Запрещенные файлы для изменения

Указать, какие файлы Codex не должен менять без отдельного разрешения.

Пример:

```text
Запрещено изменять:
- System_Map.md;
- System_Component_Registry.md;
- Agent_Registry.md;
- Data_Ownership_Map.md;
- Document_Placement_Rules.md;
- общую структуру репозитория;
- unrelated services;
- unrelated integrations;
- unrelated prompts;
- roadmap.
```

Пример для конкретного сценария:

```text
Запрещенные файлы:
- docs/architecture/System_Map.md
- docs/architecture/System_Component_Registry.md
- knowledge/agents/Agent_Registry.md
- docs/data/Data_Ownership_Map.md
- docs/roadmap/*
```

---

## Финальное правило

Use case должен быть достаточно конкретным, чтобы Codex мог понять задачу.

Но use case не должен превращаться в низкоуровневое техническое задание.

Сначала описывается пользовательский сценарий и границы.

После этого можно создавать отдельную задачу на реализацию.
