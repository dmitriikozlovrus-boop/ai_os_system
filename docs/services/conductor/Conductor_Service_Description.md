# Conductor Service Description

## Назначение документа

Этот документ описывает текущий MVP-сервис `Conductor / Дирижёр` в AI OS.

Документ фиксирует:

- что такое `Conductor`;
- какую роль он выполняет;
- как он связан с `Lyuba`;
- что входит в его ответственность;
- что не входит в его ответственность;
- какие внутренние модули предполагаются;
- какие документы должен читать Codex перед изменением логики.

Документ не является roadmap и не описывает будущую целевую реализацию в деталях.

Операционные детали MVP должны храниться отдельно:

```text
docs/services/conductor/Conductor_MVP_Operations.md
```

## Краткое описание

`Conductor / Дирижёр` — текущий рабочий MVP-сервис AI OS.

Он принимает входящую информацию, классифицирует ее, создает и обновляет записи, синхронизирует задачи и возвращает результат обработки.

На текущем этапе код сервиса находится в корневой папке:

```text
conductor/
```

Целевая архитектура может предусматривать будущий перенос в:

```text
apps/conductor/
```

Такой перенос не выполняется без отдельной миграционной задачи.

## Роль в архитектуре AI OS

`Conductor` является текущей реализацией оркестрационного слоя AI OS.

В логической архитектуре он относится к:

```text
Orchestration Layer
```

Его задача — не выполнять все действия самостоятельно, а координировать движение входящей информации между слоями системы.

Типовая логика:

```text
Input
 ↓
Conductor
 ↓
Core Logic
 ↓
Intelligence Layer
 ↓
Integration Layer
 ↓
Data Layer / Execution Layer
 ↓
Result / Feedback
 ↓
Output Interface
```

`Conductor` должен работать независимо от конкретного пользовательского интерфейса. `Lyuba` сейчас является основным интерфейсом для части сценариев, но не должна быть единственным возможным входом или выходом системы.

## Связь с Lyuba

`Lyuba` — Telegram-бот и один из интерфейсов взаимодействия пользователя с AI OS.

`Conductor` — оркестратор обработки.

Правильное разделение:

```text
Lyuba = Input / Output Interface
Conductor = Orchestration Service
```

Типовой маршрут:

```text
User
 ↓
Lyuba / Telegram
 ↓
Conductor
 ↓
Core Logic / Integrations
 ↓
Notion / Todoist / OpenAI
 ↓
Result
 ↓
Lyuba / Telegram
 ↓
User
```

`Lyuba` принимает сообщение и возвращает результат пользователю.

`Conductor` решает, что с сообщением делать.

## Текущие MVP-функции

На текущем этапе `Conductor` участвует в следующих процессах:

- прием входящих сообщений через Telegram webhook;
- обработка текстовых сообщений;
- обработка голосовых и аудиосообщений;
- транскрибация аудио через OpenAI;
- AI-классификация входящей информации;
- определение, является ли вход задачей, вопросом на изучение или смешанным входом;
- создание задач в Notion `Tasks`;
- создание записей в Notion `Study / На изучение`;
- уточнение недостающих данных через Telegram;
- локальное хранение ожидающих уточнений;
- двусторонняя синхронизация `Tasks DB` и Todoist;
- возврат результата пользователю;
- обработка ошибок и спорных случаев в рамках MVP.

Подробности запуска, webhook, переменных окружения, Notion, Todoist, OpenAI transcription и команд должны быть вынесены в `Conductor_MVP_Operations.md`.

## Ответственность Conductor

`Conductor` отвечает за:

- прием входящего сигнала от интерфейсов или источников;
- определение типа входа;
- классификацию информации;
- определение целевой сущности;
- выбор сценария обработки;
- передачу данных в нужный блок `Core Logic`;
- вызов `Intelligence Layer` при необходимости;
- выбор нужной интеграции;
- инициирование создания или обновления записей;
- запуск действий во внешних системах через интеграции;
- возврат результата в подходящий output-канал;
- фиксацию ошибки или спорного случая;
- соблюдение правил маршрутизации и источников истины.

## Что не входит в ответственность Conductor

`Conductor` не отвечает за:

- роль пользовательского интерфейса;
- замену `Lyuba`;
- хранение всех данных как источник истины;
- выполнение функций Notion, Todoist, Gmail, Calendar или других API;
- прямое смешивание всей бизнес-логики в одном файле;
- самостоятельное изменение архитектуры AI OS;
- автоматическое создание новых агентов;
- автоматическое изменение структуры БД без обновления документации;
- принятие roadmap-решений;
- реализацию всех будущих use cases;
- выполнение функций специализированных агентов без отдельного описания.

## Предполагаемые внутренние модули

На уровне логики `Conductor` может быть разделен на несколько внутренних блоков.

Это не обязательно текущая физическая структура кода. Это логическое разделение, которым нужно руководствоваться при развитии сервиса.

| Модуль | Назначение |
|---|---|
| Input Handler | прием входящих сообщений и событий |
| Telegram Handler | обработка Telegram webhook и Telegram-ответов |
| Transcription Handler | обработка аудио и голосовых сообщений |
| Classification Logic | определение типа входа и целевой сущности |
| Entity Routing Logic | выбор целевой БД или сценария обработки |
| Field Extraction Logic | извлечение полей из входящего сообщения |
| Validation Logic | проверка обязательных полей и качества данных |
| Clarification Logic | запрос уточнений у пользователя |
| Notion Integration | создание и обновление записей в Notion |
| Todoist Integration | синхронизация задач с Todoist |
| Sync Logic | правила двусторонней синхронизации |
| Error Handling | обработка ошибок и спорных случаев |
| Result Formatter | подготовка ответа пользователю |

## Связь с первым use case

Первый целевой use case:

```text
правильное раскладывание сущностей по БД с правильным набором данных
```

Для него `Conductor` должен координировать:

```text
Input
 ↓
Entity Classification
 ↓
Field Extraction
 ↓
Routing Rules
 ↓
Target DB
 ↓
Validation
 ↓
Result / Error
```

На этом этапе не требуется создавать отдельного агента для каждой сущности.

Основной фокус:

- `Conductor`;
- `Core Logic`;
- `Entity Classification`;
- `Field Extraction`;
- `Routing Rules`;
- `Validation`;
- `Notion Integration`;
- `Error Log`.

## Связь со вторым use case

Второй целевой use case:

```text
двусторонняя интеграция Tasks DB ↔ Todoist
```

Для него важно разделять:

```text
Conductor = координация
Task Logic = правила задач
Todoist Integration = техническая связь с Todoist
Data Ownership Map = источник истины
Sync Rules = правила синхронизации
Conflict Rules = правила конфликтов
Error Log = фиксация сбоев
```

`Task Agent` не является обязательным для реализации базовой синхронизации.

## Правила для Codex

Перед изменением логики `Conductor` Codex должен определить:

1. Меняется ли архитектура или только реализация.
2. Какой use case затрагивается.
3. Какие сущности участвуют.
4. Какие источники истины используются.
5. Какие интеграции затрагиваются.
6. Какие файлы разрешено менять.
7. Какие файлы запрещено менять.

Codex не должен:

- переносить `conductor/` в `apps/conductor/` без отдельной миграционной задачи;
- менять архитектурные документы без явного запроса;
- создавать новых агентов без обновления `Agent_Registry.md`;
- менять источники истины без обновления `Data_Ownership_Map.md`;
- смешивать Telegram-интерфейс, Conductor, Core Logic и Integrations;
- превращать README сервиса в операционный мануал.
