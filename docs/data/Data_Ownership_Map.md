# Карта владения данными

## Назначение документа

Этот документ фиксирует, где находится источник истины для каждого типа данных в системе ИИ-ОС.

Документ нужен для того, чтобы Codex, агенты и будущие сервисы не принимали самостоятельных решений о том, где должны жить данные.

Если возникает вопрос, куда записывать, откуда читать или где обновлять данные, сначала нужно смотреть этот файл.

---

## Главное правило

Каждый тип данных должен иметь явно определённый источник истины.

Источник истины — это место, где хранится основная, наиболее полная и авторитетная версия данных.

Другие системы могут быть интерфейсами, зеркалами, каналами исполнения или источниками сырого входящего сигнала, но они не становятся источником истины автоматически.

---

## Базовые принципы

1. Codex не должен сам решать, где живут данные.
2. Если тип данных уже указан в этом документе, нужно использовать указанное место хранения.
3. Если тип данных не указан, Codex должен зафиксировать вопрос, а не создавать новое место хранения самостоятельно.
4. Если одна сущность используется в нескольких системах, нужно различать:
   - основной источник истины;
   - рабочий интерфейс;
   - зеркало;
   - внешний источник входящих данных;
   - канал результата.
5. Прямые ссылки на базы данных Notion в этом документе пока не используются.
6. Для текущего этапа достаточно указывать логическое имя базы данных.
7. Конкретные ссылки, идентификаторы баз и технические параметры должны быть вынесены в отдельный реестр баз данных, когда структура стабилизируется.

## Роль Notion

Notion на текущем этапе является основным рабочим хранилищем пользовательских сущностей.

В Notion живут:

- задачи;
- события;
- исследования;
- проблемы;
- идеи;
- книги;
- фильмы;
- товары;
- коммуникации;
- системные ошибки;
- улучшения.

Notion используется как рабочая база данных, а не как источник устойчивых правил системы.

Устойчивые правила, архитектурные принципы, описания сущностей и стандарты разработки должны храниться в GitHub.

---

## Роль Todoist

Todoist является интерфейсом исполнения задач.

Todoist нужен для:

- визуального управления задачами;
- ежедневного исполнения;
- быстрых списков;
- синхронизации задач в рабочем режиме.

Todoist не является полным источником истины по задачам.

Если данные задачи отличаются в Notion и Todoist, нужно смотреть на назначение поля:

- структурные поля задачи хранятся в Notion;
- статус ежедневного исполнения может отражаться в Todoist;
- итоговая логика синхронизации должна быть описана отдельно в документации по данным.

---

## Роль Google Calendar

Google Calendar является рабочим календарём и источником истины по фактическому времени календарных событий.

Google Calendar хранит:

- дату;
- время начала;
- время окончания;
- повторяемость;
- участников;
- календарный статус события.

Notion хранит расширенный контекст события:

- связи с проектами;
- связи с задачами;
- итоги;
- последующие действия;
- аналитические комментарии;
- источник возникновения события.

---

## Роль Gmail

Gmail является источником входящих и исходящих писем.

Gmail не является основной базой коммуникаций в ИИ-ОС.

Если письмо превращается в структурированную коммуникацию, задачу, событие, исследование или проблему, итоговая запись должна быть создана в соответствующей базе Notion.

---

## Роль Telegram и Любы

Люба — это Telegram-интерфейс пользователя.

Люба может использоваться для:

- быстрого ввода информации;
- получения результата;
- уточнений;
- подтверждений;
- обратной связи;
- запуска сценариев.

Люба не является источником истины по данным.

Telegram-сообщение является входящим сигналом. После обработки оно должно быть преобразовано в нужную сущность и записано в правильный источник истины.

---

## Роль GitHub

GitHub является источником истины для:

- архитектуры системы;
- правил системы;
- описаний сущностей;
- правил классификации;
- стандартов разработки;
- описаний агентов;
- описаний сервисов;
- технической документации;
- продуктовой документации;
- решений по развитию системы.

GitHub не является основной пользовательской базой данных.

Пользовательские записи живут в Notion, Todoist, Google Calendar, Gmail и других рабочих системах в соответствии с этой картой.

---

## Роль OpenAI и моделей ИИ

Модели ИИ не являются источником истины.

Модели могут:

- классифицировать входящую информацию;
- извлекать сущности;
- предлагать структуру;
- создавать черновики;
- проверять качество;
- формировать выводы;
- помогать с маршрутизацией;
- предлагать исправления.

Модель не должна самостоятельно менять источник истины без правила, описанного в документации.

---

## Правило для Codex

Codex обязан использовать этот документ перед созданием, изменением или переносом данных.

Codex не должен:

- создавать новую базу данных без явного указания;
- переносить сущность в другую систему без правила;
- считать интерфейс источником истины;
- считать зеркало главным хранилищем;
- смешивать пользовательские данные и системные правила;
- класть устойчивые правила в Notion;
- класть временные рабочие записи в GitHub.

Если нужного типа данных нет в этом документе, Codex должен зафиксировать вопрос человеку.

## Пограничные случаи

### Задача из Telegram

Входящий сигнал приходит через Любу.

Итоговая задача должна быть создана в БД задач Notion.

Todoist используется как интерфейс исполнения.

### Встреча из Gmail

Письмо находится в Gmail.

Если письмо содержит встречу, событие должно быть создано в Google Calendar и связано с БД событий Notion.

Gmail остаётся источником исходного письма, но не становится базой событий.

### Исследование из идеи

Идея сначала хранится в БД идей Notion.

Если она превращается в вопрос для изучения, создаётся запись в БД исследований Notion.

Исходная идея может быть связана с исследованием.

### Ошибка в работе сервиса

Сырой технический лог может находиться в файлах логирования или во внешнем сервисе.

Структурированная ошибка должна быть записана в БД системных ошибок Notion.

Если ошибка требует изменения архитектуры, соответствующее решение должно быть оформлено в GitHub.

### Новое правило классификации

Если появляется устойчивое правило классификации, оно должно быть записано в GitHub в раздел `knowledge/classification/`.

Notion может содержать рабочие заметки, но не является источником истины для правил классификации.

---

## Что не нужно делать на текущем этапе

Пока не нужно:

- вставлять прямые ссылки на базы Notion;
- фиксировать технические идентификаторы баз;
- создавать отдельную физическую схему хранения для каждой сущности;
- преждевременно переносить данные в PostgreSQL или Supabase;
- усложнять карту владения данными до полноценной схемы синхронизации.

На текущем этапе задача документа — зафиксировать, где находится главный источник истины для каждого типа данных.
